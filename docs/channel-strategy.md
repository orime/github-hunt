# GitHub Hunt 内容生产与商业化路径

## 产品顺序

先把网站做成主产品，再把同一份日更数据分发到公众号、短视频、小程序，不同时开四条研发线。

1. **网站**：每天固定时间刷新，沉淀归档与品牌。
2. **公众号**：复用 `outputs/wechat/` 的 Markdown 草稿，人工精修后发布。
3. **短视频**：复用 `outputs/video/` 的 60-90 秒脚本，做周更或精选版日更。
4. **小程序**：复用 `outputs/miniapp/` 的卡片流，做只读消费端，而不是主站替代品。

## 为什么这样排

网站负责沉淀，公众号负责分发，视频负责放大，小程序负责承接微信生态内消费。这样一套数据，四种出口，但只有一个真实底座。

## 当前仓库里已经落下的路径

- 网站数据：`web/public/data/repos.daily.json`
- 公众号草稿：`outputs/wechat/daily-YYYY-MM-DD.md`
- 视频脚本：`outputs/video/daily-YYYY-MM-DD.md`
- 小程序卡片：`outputs/miniapp/daily-YYYY-MM-DD.json`

## 最现实的商业化顺序

先卖可信分发位，不先卖订阅。

1. **赞助位 / 工具位**：站内和公众号文末挂合作位。
2. **招聘位**：把开发者岗位放进日报或站内独立模块。
3. **合作伙伴页**：为云厂商、开发工具、AI 产品留入口。
4. **后续再做**：等流量稳定后，再上广告、会员、细分付费订阅。

## 内容生产规则

- 日更内容优先保证可信度，不做全自动直发。
- 先机器生成草稿，再人工 5 分钟审一遍。
- 正文不卖榜单，只卖明确标注的赞助位。
